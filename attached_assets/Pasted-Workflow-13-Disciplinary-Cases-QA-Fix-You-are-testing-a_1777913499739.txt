You are testing and fixing the disciplinary workflow end-to-end.

Scope:
- Frontend route: /app/disciplinary
- Disciplinary cases list
- Violation types page
- Create new case form
- Case status flow
- Role permissions
- Notifications
- Audit logs
- DB persistence

Run full discovery first before changing code:
1. Inspect DB schema/tables for disciplinary cases and violation types.
2. Inspect backend API routes related to disciplinary/violations.
3. Inspect frontend component/service/template for /app/disciplinary.
4. Compare frontend payload fields with backend expected fields and DB columns.
5. Identify stub/hardcoded endpoints, field mismatches, missing routes, missing role guards, missing joins, missing audit, missing notifications.

Required accounts to test:
- hr
- manager.hr
- employee01
- employee02
- payroll

Expected permissions:
- HR/admin/superadmin can create, view, edit, investigate, close, and manage violation types.
- Manager may view only team-related cases if the business logic supports manager access; otherwise block with 403. Do not allow manager to manage global violation types unless explicitly intended.
- Employee can only see their own disciplinary cases, if employee self-view exists. Employee must NOT create, edit, delete, close, or manage violation types.
- Payroll must be blocked from disciplinary management and sensitive employee disciplinary data unless explicitly intended.
- No unauthenticated access.

Test and fix these areas:

A) Violation Types
- GET violation types must return real DB rows, not stub data.
- POST create violation type must validate required fields:
  - code
  - nameAr
  - nameEn
- Duplicate code must return 409, not 500.
- PUT/PATCH update must persist changes.
- DELETE should soft-delete or deactivate if used by cases; must not break existing case history.
- Only HR/admin/superadmin can manage types.
- Employee, manager, payroll must get 403 for create/update/delete.
- Response shape must match frontend fields:
  - id
  - code
  - nameAr
  - nameEn
  - isActive/status
  - createdAt/updatedAt if used

B) Disciplinary Cases — Create
Open a new case through API and verify UI-compatible payload.
Validate required fields:
- employeeId
- violationTypeId or violationType/code depending frontend
- violationDate
- description
- proposedPenalty / action / penaltyType if required by UI
- amount only if penalty requires amount
- notes optional

Reject:
- missing employee
- missing violation type
- missing description
- invalid violationDate
- future violationDate if not allowed
- negative amount
- employee from another company
- invalid violation type
- unauthenticated request
- unauthorized role

Expected create result:
- Case inserted in DB.
- companyId saved from JWT, not body.
- createdBy/createdById saved from JWT.
- Initial status is open or draft depending button:
  - “Save as draft” should create draft status.
  - “Open case” should create open/investigation-ready status.
- Activity log written.
- Employee and/or manager notification created only if product logic expects it. If notification is designed, route it precisely, not broadcast.

  - violationTypeNameAr/En

  - createdAt
  - updatedAt
  - createdBy

- Manager sees only team cases, or 403 if manager access is not intended.

  - search employee/description

- Summary counters must be real:
  - total cases

D) Case Detail + Status Flow
Test and fix endpoints/actions if frontend calls them:
- GET /api/disciplinary/:id or equivalent
- update case
- start investigation
- close/resolve case
- reject/cancel case
- delete/soft-delete case
- save draft
- open draft

- draft → open

- under_investigation → resolved/closed

- cannot close without final decision/penalty if required
- cannot modify closed cases except allowed admin correction
- double-close returns 409
- invalid transition returns 409 or 400, not silent success
