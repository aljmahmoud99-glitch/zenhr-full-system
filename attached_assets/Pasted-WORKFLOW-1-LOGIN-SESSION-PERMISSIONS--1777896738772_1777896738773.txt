TASK 1 — IDENTIFY THE LOGIN SYSTEM
═══════════════════════════════════
Before testing, read and report:

1. Frontend:
   - What is the login route? (e.g., /login, /auth/login)
   - What component handles it? (filename + path)
   - What HTTP call does it make on submit? (method + URL + body shape)
   - Where is the token stored after login? (localStorage key name)
   - Where is the user object stored? (localStorage key? in-memory service? both?)

2. Backend:
   - What is the login API endpoint? (POST /api/auth/login or similar)
   - What does a successful response look like? ({ token, user } or { accessToken, refreshToken, ... }?)
   - What claims are inside the JWT token? (decode and list: userId, employeeId, role, companyId, etc.)
   - Is there a refresh token mechanism? If yes: how is it used?

3. Permission system:
   - How does the frontend determine what a user can see?
   - Is there a permissions API call after login? (GET /api/permissions/my or similar)
   - Or does the frontend derive permissions from the role string alone?
   - Is there an Angular AuthGuard? What does it check?
   - Is there a RoleAccessService or PermissionService?

Report all findings as a structured list before running any tests.

TASK 3 — JWT TOKEN VALIDATION
═══════════════════════════════════
After logging in as each role, decode the JWT token (without verifying signature — just parse the payload):

For EACH of these accounts: admin, hr, payroll, manager.it, employee01

Decode token and verify it contains:
  - userId or id (number)
  - employeeId (number — important for data scoping)
  - role (string — must match expected value)
  - companyId (number — critical for multi-tenant isolation)
  - exp (expiry timestamp — must be in the future)

Report the decoded payload for each account.
Flag any account where:
  - employeeId is missing or null
  - companyId is missing or null
  - role does not match expected

These are HIGH PRIORITY bugs — they will break every other workflow.

═══════════════════════════════════
TASK 9 — FIX ANY BUGS FOUND
═══════════════════════════════════
For each bug found in Tasks 1–8:

Before fixing:
  - Classify the bug type (see list below)
  - Identify the exact file and line causing it
  - Describe the minimal fix

Bug types:
  AUTH_MISSING        — endpoint accessible without authentication
  ROLE_LEAK           — role can access route/data it should not
  TOKEN_MISSING       — JWT missing required claim (employeeId, companyId, etc.)
  SCOPE_MISSING       — data not filtered by company_id or employee scope
  SESSION_BROKEN      — session lost on refresh, or logout does not clear correctly
  NAV_VISIBLE         — sidebar shows items the role should not see
  NAV_MISSING         — sidebar does not show items the role should see
  REDIRECT_BROKEN     — wrong redirect after login or logout
  VALIDATION_MISSING  — frontend allows empty form submission to backend

Apply only MINIMAL fixes:
  - Wrong redirect: change the route string in one place
  - Missing claim in JWT: add the claim to JwtService in one location
  - Missing route guard: add canActivate to the route definition
  - Wrong sidebar items: correct the roles array for that nav item

DO NOT refactor the auth system. DO NOT change the JWT structure beyond adding missing claims.
DO NOT touch anything not directly causing a confirmed bug.
