TASK 2 — SUBMISSION CHAIN TEST
═══════════════════════════════════
Log in as employee01.

Step 2.1 — Navigate to leave screen:
  - Route loads without 404 or 500
  - Leave balance summary visible
  - Leave request list visible (may be empty initially)

Step 2.2 — Submit a leave request:
  - Click "طلب إجازة" or equivalent
  - Select leave type: Annual Leave / الإجازة السنوية
  - Select start date: next working day
  - Select end date: 3 days after start
  - Add reason: "اختبار نظام QA"
  - Click submit

Step 2.3 — POST API validation:
  - Confirm network request: POST /api/leave/requests (or similar)
  - Status code: expected 200 or 201
  - Response body: should contain { success: true, data: { id, employeeId, status: "pending", ... } }
  - If status code ≠ 200/201: classify as API_ERROR

Step 2.4 — Database check:
  - Query DB: SELECT * FROM leave_requests WHERE employee_id = [employee01_id] ORDER BY created_at DESC LIMIT 1
  - Confirm: id exists, employee_id correct, company_id correct, status = 'pending', dates correct, created_at populated
  - If record missing from DB despite 200 response: FAKE_SUCCESS bug — critical

Step 2.5 — Read API validation:
  - Call GET /api/leave/requests (as employee01)
  - Confirm the newly created request appears in the response
  - If missing: classify as READ_MISMATCH (DB has it but GET doesn't return it)

Step 2.6 — UI rendering:
  - After submission, request must appear in the leave requests list
  - Status badge must show: "قيد الانتظار / Pending"
  - If request not visible: classify the bug (filter bug? mapping bug? cache issue?)

Step 2.7 — Balance check:
  - After submission: pending_days in leave_balances should increase by 3
  - Call GET /api/leave/balances for employee01
  - Confirm pending_days updated
  - If NOT updated: BALANCE_NOT_UPDATED bug
