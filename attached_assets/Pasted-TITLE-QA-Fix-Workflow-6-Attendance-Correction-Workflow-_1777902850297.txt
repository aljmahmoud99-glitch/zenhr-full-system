- Check-in time appears in summary.
- Toast/success message appears.

- date = today
- clock_in/check_in populated
- status = present or late
- late_minutes populated, even if 0
- attendance_type populated
- created_at populated

Audit validation:
Query activity_logs latest:
- action/type includes attendance_check_in
- description references employee01 or attendance record

Security validation:

{ employeeId: employee02_id }

Expected:
- backend ignores employeeId from body
- record remains employee01 only
- no record created for employee02

If employee02 record is created:
Type: SCOPE_BUG — critical.

TASK 9 — CORRECTION VALIDATION RULES
═══════════════════════════════════
As employee01 test invalid correction submissions:

1. Missing date.
Expected: blocked, no DB row.

2. Missing reason.
Expected: blocked, no DB row.

3. Check-out before check-in.
Expected: blocked, no DB row.

4. Future date.
Expected: blocked unless policy allows future corrections.

5. Duplicate pending correction for same date/type.
Expected: conflict / validation error.

6. employee01 attempts correction for employee02 by sending employeeId in body.
Expected:
- backend ignores employeeId or rejects
- correction belongs to employee01 only.

If invalid request creates DB row:
Type: VALIDATION_MISSING.
