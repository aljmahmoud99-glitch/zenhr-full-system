PART 2 — DEFINE ATTENDANCE STATE MACHINE
═══════════════════════════════════
Implement or enforce this attendance state model.

For each employee per date, there should be at most ONE primary attendance record.

Statuses:
- pending / not_started: no check-in yet
- present: checked in, not checked out
- completed: checked in and checked out
- absent: no check-in by end of day or manually marked
- late: checked in after allowed grace period
- corrected: record updated through approved correction
- cancelled / invalid: if record is voided by HR

Rules:
1. Employee cannot check in twice on same date.
2. Employee cannot check out without check-in.
3. Employee cannot check out twice.
4. Check-out time must be after check-in time.
5. Work hours must be calculated on backend.
6. Late flag must be calculated on backend using configured shift/start time if available.
7. If shift config is missing, use safe default:
   - work start: 09:00
   - grace period: 15 minutes
   - work day: 8 hours
8. Never rely on frontend calculation for final stored values.

Expected attendance record fields returned to UI:
{
  id,
  employeeId,
  employeeName,
  employeeCode,
  departmentName,
  date,
  checkInTime,
  checkOutTime,
  workHours,
  status,
  isLate,
  lateMinutes,
  attendanceType,
  locationName,
  correctionStatus,
  correctedAt,
  correctedByName,
  createdAt,
  updatedAt
}

If DB has different column names, map them consistently in API responses.

PART 3 — FIX CHECK-IN / CHECK-OUT BACKEND
═══════════════════════════════════
Fix or add these endpoints.

Employee endpoints:
POST /api/attendance/check-in
POST /api/attendance/check-out
GET  /api/attendance/my-today
GET  /api/attendance/my-records

Required check-in behavior:
1. Auth required.
2. user.employeeId required.
3. Use employeeId from JWT only.
4. companyId from JWT if table supports it.
5. Date = today server-side unless explicitly supported.
6. If today's record already has checkInTime:
   return 409 with message "Already checked in".
7. Create or update today's attendance record:
   - employeeId
   - date
   - checkInTime = now
   - status = present or late
   - isLate
   - lateMinutes
   - attendanceType = office/default
   - createdAt/updatedAt
8. Create activity log:
   action = attendance_check_in
   entityType = attendance
   entityId = record.id
   actorUserId = current user
9. Return { success:true, data: normalizedAttendanceRecord }

Required check-out behavior:
1. Auth required.
2. user.employeeId required.
3. Find today's attendance record for current employee.
4. If no check-in: return 400.
5. If already checked out: return 409.
6. Set checkOutTime = now.
7. Calculate workHours = decimal hours rounded to 2 decimals.
8. Set status:
   - completed if not late
   - late if late but checked out
   or preserve status late with completed flag if schema supports.
9. Create activity log:
   action = attendance_check_out
10. Return normalized record.

GET /api/attendance/my-today:
Must return today's record for current employee only.
If no record:
return state:
{
  status: "pending",
  checkInTime: null,
  checkOutTime: null,
  workHours: null,
  isLate: false,
  lateMinutes: 0
}

GET /api/attendance/my-records:
Must return current employee records only.

PART 5 — FIX CORRECTION REQUESTS LOGIC
═══════════════════════════════════
Correction requests must be linked to attendance records where possible.

Correction types:
- missing_check_in
- missing_check_out
- time_correction
- absence_correction
- location_correction

Expected correction request fields:
{
  id,
  employeeId,
  employeeName,
  attendanceId,
  correctionType,
  date,
  currentCheckInTime,
  currentCheckOutTime,
  requestedCheckInTime,
  requestedCheckOutTime,
  reason,
  status,
  managerApprovedById,
  managerApprovedAt,
  hrApprovedById,
  hrApprovedAt,
  rejectionReason,
  createdAt,
  updatedAt
}

Employee create endpoint:
POST /api/attendance/corrections
or existing frontend path if different.

Required create behavior:
1. Auth required.
2. employeeId from JWT only.
3. Date required.
4. Reason required.
5. At least one requested time/change required.
6. Find existing attendance record for employee + date if present.
7. Store attendanceId if found.
8. Store current values snapshot:
   - currentCheckInTime
   - currentCheckOutTime
9. Store requested values.
10. status = pending.
11. Create notifications:
   - direct manager
   - HR if current policy requires immediate visibility
12. Create activity log:
   action = attendance_correction_requested.
13. Return created request.

Validation:
- requestedCheckOutTime must be after requestedCheckInTime if both provided.
- Cannot create duplicate pending correction for same employee/date/type.
- Employee cannot create correction for another employee.
- Employee cannot correct future dates unless policy explicitly allows it.

Manager approval:
PUT /api/attendance/corrections/:id/approve
or existing route if different.

Behavior:
1. manager can approve only direct reports / scoped team.
2. pending → manager_approved.
3. Notify HR.
4. Do NOT apply correction to attendance record yet unless current policy says manager-only approval.
5. Activity log action = attendance_correction_manager_approved.

HR final approval:
If request status = manager_approved or pending if HR bypass allowed:
1. Apply requested correction to linked attendance record.
2. If attendance record does not exist, create one for employee/date.
3. Recalculate:
   - workHours
   - status
   - isLate
   - lateMinutes
4. Set correctionStatus = approved/corrected if schema supports.
5. Mark request approved.
6. Store hrApprovedById/hrApprovedAt.
7. Notify employee.
8. Activity log:
   - attendance_correction_hr_approved
   - attendance_record_corrected
9. Return updated request + updated attendance record.

Reject:
1. manager/hr can reject according to scope.
2. Save rejectionReason.
3. status = rejected.
4. Notify employee.
5. Do not modify attendance record.
6. Activity log action = attendance_correction_rejected.

Security:
- Employee cannot approve/reject.
- Employee cannot see others' correction requests.
- Wrong manager cannot approve/reject.
- Approved/rejected/cancelled correction cannot be approved again.
